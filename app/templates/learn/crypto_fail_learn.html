Cryptographic Failures

Cryptographic failures are vulnerabilities where sensitive data like passwords, credit card numbers, or personal information is not properly protected. This could be due to the web application using bad encryption, no encryption, or misusing encryption tools.

Attackers can now target, steal and misuse this unprotected data. This is the root cause of sensitive data exposure.

The Danger

If encryption systems of an application are weak or used incorrectly, attackers can do things including but not limited to:

  • Break the encryption and steal private data.
  • Cause data breaches, identity theft or fraud.
  • Obtain unauthorized access to user accounts.

How they occur

Here are potential causes of cryptographic failures:

  • Using old or broken encryption algorithms.
  • Sending data over the internet without encryption (such as over HTTP).
  • Storing passwords using weak protection.
  • Reusing or poorly managing encryption keys.
  • Not checking security certificates properly.
  • Letting programming flaws leak sensitive data that can be exploited.

Preventing Cryptographic Failures

Preventing Cryptographic Failures is a task that requires multiple steps. Some include:

Knowing What Needs Protection

  • Identify the sensitive data your app collects.
  • Protect it in accordance to regulations (like GDPR) or standards (like PCI-DSS).

Avoiding Common Mistakes

  • Don’t send sensitive data in plain text over the internet (use HTTPS, not HTTP).
  • Don’t use outdated or weak encryption tools.
  • Don’t reuse encryption keys or store them in code.
  • Don’t skip certificate checks (i.e. make sure that secure connections are truly secure).
  • Don’t use insecure random number generators or insecure encryption settings.

Using Proper Encryption

For this step, ensure that:

  • Encrypt all sensitive data at rest (when stored) and in transit (when sent).
  • Use strong, up-to-date encryption algorithms and tools.
  • Enforce HTTPS with HSTS headers.
  • Never store passwords in plain text. Use strong hashing such as:
    • bcrypt
    • scrypt
    • Argon2
    • PBKDF2
  • Avoid insecure hash functions (like MD5 or SHA1).
  • Use authenticated encryption to keep data both encrypted and verified.

For more information and detailed explanations about this topic, please look at the optional further reading linked below.

Example Attack Scenarios

Scenario 1: A site stores encrypted credit card numbers but decrypts them automatically when retrieved. If a hacker finds an SQL injection flaw, they could get the card numbers in plain text.

Scenario 2: A website doesn’t use HTTPS. A Hacker on public Wi-Fi can spy on users, steal their login cookies, and hijack their accounts.

Scenario 3: Passwords are stored with weak or no hashing. A file upload flaw allows an attacker to retrieve the password database. A hacker uploads a file, gets access to the password database, and uses pre-made tables (rainbow tables) to crack all the passwords quickly.
