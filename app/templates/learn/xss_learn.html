Cross-site scripting (XSS)

Cross-site scripting (XSS) is a type of vulnerability found in some websites or web applications. It allows attackers to add harmful code (or scripts) into a web page that is viewed by other users.
This can lead to harmful outcomes such as an attacker stealing personal data and changing the content that users see on a page.

Understanding the Vulnerability

To understand this vulnerability, imagine a website asking for a comment (or any other input). Because the page has an XSS vulnerability, this input is used within the output page generated without validation. Instead of a comment, an attacker could "trick" the website into including a malicious script when they submit the form (i.e., the comment here). When the page is now opened by a user, the malicious script runs without their knowledge.

The user’s browser has no way to know that the script should not be trusted and will execute the script. Because it thinks the script came from a trusted source, the malicious script can access any cookies, session tokens, or other sensitive information retained by the browser and used with that site. These scripts can even rewrite the content of the HTML page.

Types of XSS Attacks

There are three main types of XSS attacks, namely:

Reflected XSS

This happens when the web application takes input (like something you type) and includes this in the response without checking if it's safe.
If an attacker sends a link that includes a harmful script, it could run on the user's browser as soon as it is clicked. At that point, the script can carry out any action and retrieve any data to which the user has access.

Stored XSS

In this case, the harmful script is saved in the website’s database. On visiting the page where the script is saved, it automatically runs in the user's browser. It can affect all users who visit the page.
This is the most dangerous type of XSS because the attacker doesn't need to trick a user into clicking a link; they just need to add the script to the website once.

DOM-based XSS

Here, the vulnerability exists in client-side code rather than server-side code. It happens when the website uses JavaScript on the client side (such as in your browser) to show data. If this JavaScript processes untrusted data, it could write harmful content to the page. The attack happens without even needing the server to be involved.

Why is it dangerous?

As discussed above, XSS make affected websites unsafe for users. The browser trusts the malicious scripts as if they are from the website itself. This enables attackers to steal sensitive information, impersonate users, and change the contents of web pages, among other malicious outcomes.

Protecting from XSS

To prevent this problem, developers need to ensure that any input is properly checked and cleaned before being displayed on a page. This can be done by encoding or escaping special characters and ensuring that scripts can’t be run from user inputs.
